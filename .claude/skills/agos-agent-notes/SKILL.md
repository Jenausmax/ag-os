# AG-OS: личные заметки агентов в vault

Этот скилл — про то, как любой агент AG-OS (мастер или sub-агент) оставляет
след своей работы в Obsidian vault, чтобы знание не умирало вместе с
сессией. Потом мастер (через `agos-vault-curator`) раз в сутки курирует
эти черновики в `wiki/`.

## Когда писать

**Пиши, если:**
- пользователь задал содержательный вопрос и ты дал нетривиальный ответ
  (не «привет» и не «покажи список»);
- ты нашёл факт / ссылку / кусок кода, который пользователю было важно
  узнать;
- ты провёл исследование (web search, чтение кода, разбор логов) и
  пришёл к выводу;
- ты обнаружил баг / странность / несоответствие в проекте или внешнем
  сервисе;
- ты сделал наблюдение о пользователе, его предпочтениях или о том как
  лучше с ним работать (это уходит в memory, но для истории — заметка).

**НЕ пиши, если:**
- взаимодействие было чисто механическим (`ls`, `agent list`, эхо);
- ты просто подтвердил выполнение команды без новой информации;
- заметка будет дублировать то что уже есть в `raw/me/` за последний час
  (проверь `ls -t` перед записью);
- это чувствительные данные (токены, пароли, credentials) — их в vault
  нельзя никогда.

Лучше не написать пустую заметку, чем захламлять raw.

## Куда писать

Путь зависит от того, где ты запущен:

- **Sub-агент в docker:** `/vault/raw/me/` (смонтирован из
  `<vault_base>/raw/<твоё_имя>` на хосте, rw).
- **Sub-агент на хосте (tmux):** `$VAULT_BASE/raw/<твоё_имя>/`. Имя
  узнаёшь из переменной окружения `AG_OS_AGENT_NAME` или из имени
  текущего tmux-окна.
- **Мастер:** `$VAULT_BASE/raw/master/`.

`$VAULT_BASE` узнай через:

```bash
VAULT_BASE=$(python main.py vault path --wiki 2>/dev/null | xargs dirname)
# fallback, если CLI недоступен (docker-контейнер):
[ -z "$VAULT_BASE" ] && [ -d /vault/raw/me ] && VAULT_BASE=/vault
```

Если ни CLI, ни `/vault/raw/me` не доступны — vault выключен,
промолчи.

## Формат заметки

Всегда markdown с YAML frontmatter — иначе curator не сможет потом
нормально обработать.

**Имя файла:** `<тип>-<YYYY-MM-DD-HHMM>-<короткий-слуг>.md`. Типы:
- `convo` — взаимодействие с пользователем
- `finding` — факт, находка, результат исследования
- `observation` — наблюдение о системе / пользователе / проекте
- `bug` — обнаруженная проблема

**Шаблон:**

```markdown
---
author: <твоё_имя_агента>
type: convo | finding | observation | bug
created: 2026-04-15T19:30:00+03:00
tags: [tag1, tag2]
user: <имя пользователя из Telegram-префикса, если применимо>
chat_id: <chat_id, если из Telegram>
---

# Короткий заголовок одной строкой

## Контекст
Что происходило, о чём был запрос пользователя, почему это вообще
случилось.

## Суть
Главное содержание: что нашёл / что решил / что ответил. Факты и
ссылки, а не пересказ разговора.

## Связанное
- Ссылки на другие заметки, тикеты, коммиты (если знаешь)
- Можно `[[wiki-link]]`, curator подхватит
```

Многословность не нужна — 5-15 строк содержательного текста это норма.
Curator потом сам решит что из этого переедет в `wiki/` и как оформить.

## Пример: после ответа в Telegram

Пользователь спросил можешь ли ты удалять tmux-окна агентов. Ты нашёл
`TmuxRuntime.destroy_agent()`, ответил в Telegram. Теперь пиши:

```bash
TS=$(date +%Y-%m-%d-%H%M)
cat > "$VAULT_BASE/raw/master/finding-$TS-tmux-destroy-agent.md" <<EOF
---
author: master
type: finding
created: $(date -Iseconds)
tags: [runtime, tmux, agent-lifecycle]
user: JenausMax
chat_id: 249402107
---

# TmuxRuntime умеет убивать окна агентов

## Контекст
JenausMax спросил умею ли я сносить окна агентов через Telegram.

## Суть
\`TmuxRuntime.destroy_agent(name)\` в runtime/tmux_runtime.py:38
вызывает \`window.kill()\` по совпадению имени окна. Тот же метод
дергается из \`AgentManager\` при destroy агента, так что через
\`python main.py agent destroy --name X\` это работает end-to-end.

## Связанное
- [[agos-agent-management]]
- runtime/tmux_runtime.py:38
EOF
```

## Пример: после исследования

Ты провёл grep по кодбазе и нашёл что prompt caching нигде не
используется. Это наблюдение — пиши `observation-...md` с соответствующим
содержимым и тегами.

## Чего НЕ делать

- **Не пиши в `raw/<чужое_имя>/`** — это их пространство, у тебя нет
  прав (в docker физически, на хосте — по соглашению).
- **Не пиши в `wiki/`** напрямую, если ты не мастер. Wiki — single-writer,
  только `agos-vault-curator` у мастера это делает.
- **Не дублируй.** Перед записью быстро проверь `ls -t $VAULT_BASE/raw/me/`
  — если за последний час уже была похожая заметка, обнови её, а не
  создавай новую.
- **Не записывай секреты** (API-ключи, токены, содержимое
  `~/.claude/credentials.json`, пароли).
- **Не блокируй ответ пользователю заметкой.** Сначала отвечай, потом
  пиши. Если уже ответил и решаешь писать или нет — склоняйся к «писать»,
  когда есть содержательный факт, и к «не писать», когда это был
  рутинный обмен.

## Связанное

- `agos-vault-curator` — мастер забирает эти черновики и курирует в
  `wiki/`. Твои заметки — сырьё для него.
- `agos-telegram-reply` — если ты мастер и только что ответил
  пользователю через telegram_reply, это хороший момент подумать о
  заметке.
