# agos-agent-recall

## Когда использовать

- Тебя только что запустил scheduler после `/clear` — у тебя пустой контекст, но задача может ссылаться на прошлые запуски.
- В prompt мастер просит "вспомнить, на чём остановился", "recall context", "use agos-agent-recall" — это явный триггер.

## Алгоритм

1. **Узнай своё имя.** Переменная окружения `AG_OS_AGENT_NAME` содержит твоё имя. Если её нет — попроси мастера её установить и продолжай без recall.

2. **Прочитай свои недавние заметки.** Загляни в `vault/agents/<AG_OS_AGENT_NAME>/`. Возьми последние ~10 файлов по времени модификации (не больше — не раздувай контекст обратно).

3. **Отфильтруй по релевантности.** Из текущего prompt-а возьми ключевые слова (тема, объект, тип задачи). Оставь те заметки, где эти слова встречаются. Если ничего не подходит — оставь 1-2 самых свежих как общий фон.

4. **Сформируй короткое мысленное резюме (НЕ выводи в чат).** Ответь себе на три вопроса:
   - Что я делал в прошлые запуски этой задачи?
   - На чём именно остановился (последний результат, открытый вопрос, ошибка)?
   - Что должно измениться или продолжиться сейчас?

5. **Только теперь — выполняй задачу.** Если по ходу выполнения обнаружится что-то новое (вывод, проблема, факт) — запиши raw-заметку через скилл `agos-agent-notes`, чтобы следующий запуск тоже её увидел.

## Деградация

- Vault недоступен / папка пуста / нет релевантных заметок — выполняй задачу как обычно, без recall. Это не ошибка, просто первый запуск или отсутствие истории.
- Если `AG_OS_AGENT_NAME` не выставлен — не пытайся угадать имя. Пропусти recall.

## Антипаттерны

- ❌ Читать ВСЕ заметки агента — раздуешь контекст обратно, и весь смысл `/clear` пропадёт.
- ❌ Дублировать в чат всё содержимое заметок — резюме держи "у себя в голове" (в reasoning), а в ответе только результат задачи.
- ❌ Делать recall перед КАЖДЫМ ad-hoc сообщением — этот скилл только для cron-тиков и явного запроса.
