# AG-OS: управление расписанием

Этот скилл — для постановки агентов на cron и управления периодическими задачами. Под капотом — APScheduler с персистентностью в SQLite (`schedule` таблица).

## Как работает планировщик

1. `AgScheduler` поднимается в `main.py bootstrap` режимах `bot`/`all`, читает задачи из таблицы `schedule` и регистрирует их в APScheduler.
2. В cron-момент вызывается `manager.send_prompt(agent_name, prompt)` — промт уходит агенту через его runtime (tmux или docker), как обычное сообщение от пользователя.
3. Результат (`success`/`error`) и `last_run` пишутся обратно в таблицу.
4. **Hot-reload** (AGOS-0029): CLI после `schedule add/rm` шлёт живому боту `SIGUSR1` (Unix) или трогает `.ag-os-reload` (Windows), бот перечитывает таблицу без рестарта.

## Команды

### Добавить задачу

```bash
python main.py schedule add \
    --cron "<5 полей cron>" \
    --agent <agent-name> \
    --prompt "<текст промта>"
```

Cron-формат: `минуты часы день месяц день_недели` (стандартный Unix). Валидируется через `CronTrigger.from_crontab` до записи в БД — невалидный упадёт с понятной ошибкой.

**Примеры cron:**

| Выражение | Значение |
|---|---|
| `0 * * * *` | каждый час в :00 |
| `*/15 * * * *` | каждые 15 минут |
| `0 9 * * 1-5` | 9:00 по будням |
| `0 0 1 * *` | полночь 1-го числа |
| `30 8,20 * * *` | 8:30 и 20:30 ежедневно |
| `0 3 * * 0` | 3:00 по воскресеньям |

### Посмотреть задачи

```bash
python main.py schedule list
# или
python main.py schedule list --json
```

Выдаёт `#id [ON|OFF] cron @agent → prompt`. ON — `enabled=1`, OFF — `enabled=0`. Используй перед добавлением, чтобы не создать дубль.

### Удалить задачу

```bash
python main.py schedule rm --id <id>
```

### Запустить задачу сейчас (вручную)

```bash
python main.py schedule run --id <id>
```

Отправляет промт агенту немедленно, не дожидаясь следующего cron-тика. Полезно для отладки или когда пользователь говорит «запусти обработку прямо сейчас».

## Типовые сценарии

### Сценарий 1: «проверяй почту раз в час»

Сначала убедись, что нужный агент существует и у него есть инструмент для чтения почты (MCP gmail, CLI-утилита, или доступ к API). Если инструмента нет — предупреди пользователя, что cron будет срабатывать, но агент честно ответит «нет инструмента».

```bash
# 1. Проверь агента
python main.py agent list | grep mail

# 2. Поставь на cron
python main.py schedule add \
    --cron "0 * * * *" \
    --agent mail-bot \
    --prompt "проверь новую непрочитанную почту за последний час и доложи если что-то важное"

# 3. Покажи пользователю что получилось
python main.py schedule list
```

### Сценарий 2: «ежедневная сводка новостей ИБ в 9 утра по будням»

```bash
python main.py schedule add \
    --cron "0 9 * * 1-5" \
    --agent researcher \
    --prompt "собери сводку ключевых новостей по информационной безопасности за последние 24 часа, приоритет — критичные CVE и крупные инциденты"
```

### Сценарий 3: «что у нас в расписании»

```bash
python main.py schedule list
```

Прочитай вывод, сделай пользователю краткую сводку на человеческом языке: сколько задач, кто какой агент делает, когда, статус последнего запуска.

### Сценарий 4: «отмени задачу которая проверяет почту»

```bash
python main.py schedule list | grep -i mail
# узнаёшь id, например #7
python main.py schedule rm --id 7
```

### Сценарий 5: «запусти сейчас обработку vault'а»

Обработка vault raw→wiki — это отдельная cron-задача, которая создаётся автоматически в bootstrap при `vault.enabled: true` (с маркером `[vault-processing]` в промте).

```bash
# Найди задачу
python main.py schedule list | grep vault-processing
# Запусти вручную
python main.py schedule run --id <id>
```

## Важные нюансы

### Задача vs агент

Планировщик — это **будильник**, а не тул. Он передаёт агенту текст промта — дальше агент делает то, что умеет. Если пользователь говорит «запусти агента по cron чтобы он писал отчёт в отчётную таблицу», убедись что:
- у агента есть доступ к этой таблице (API-ключ, MCP-сервер, файловая система);
- промт достаточно конкретный — «напиши отчёт в X» работает, «напиши отчёт» оставит агента гадать куда.

### Частота

Не ставь задачи чаще чем раз в минуту без причины. Каждый тик — это LLM-запрос через provider (если не `claude_subscription`), который стоит денег. Для большинства кейсов достаточно часа.

### Длинные промты

`--prompt "..."` в bash — экранирование может стать проблемой. Для длинных промтов сначала сохрани в файл, потом используй `--prompt "$(cat prompt.txt)"`. Или используй Telegram-команду `/schedule_add` через бот.

### Telegram vs CLI

Есть два способа ставить задачи:
- **CLI** (`python main.py schedule add`) — удобнее для мастера из его окна, hot-reload автоматом.
- **Telegram** (`/schedule_add`) — удобнее для пользователя, обходит file-system, сразу идёт через живой `AgScheduler.add_task` без reload.

Оба попадают в ту же таблицу и работают одинаково.

## Чего НЕ делать

- **Не модифицируй `scheduler/scheduler.py`** — все операции через CLI или Telegram.
- **Не пиши SQL напрямую в `schedule`** — потеряешь hot-reload, сломаешь idempotent diff.
- **Не ставь в cron задачи на тесты/отладку без tear-down.** Забытая `*/5 * * * *` с промтом «тест» будет спамить агента каждые 5 минут. Всегда ставь `dynamic`-задачи вручную через `schedule run`, а не через cron.

## Periodic tasks: автоочистка контекста и recall

Для **периодических** (cron) задач AG-OS по умолчанию очищает контекст подопечного агента перед каждым тиком (флаг `clear_before=1` в БД, дефолт CLI). Это экономит токены подписки на длинных горизонтах (агент типа `finik`, который крутится сутками).

Чтобы агент не "забыл" предысторию совсем, **обязательно** оборачивай пользовательский prompt в шаблон, явно вызывающий скилл `agos-agent-recall`:

    Используй скилл agos-agent-recall, чтобы вспомнить контекст из своих заметок в vault.

    Задача: <оригинальный prompt пользователя>

Этот шаблон применяется только к **cron-задачам**. Для разовых (one-shot, `schedule run` / прямой prompt) — не оборачивай и не выставляй clear: контекст там между вызовами полезен.

### CLI

- По умолчанию: `python main.py schedule add --cron "0 * * * *" --agent finik --prompt "<обёрнутый>"` — флаг `clear_before` в БД будет = 1.
- Отключить чистку (редко, для частых коротких задач): добавить `--no-clear`.

### Когда НЕ применять

- Разовая задача — не оборачивай и не передавай `--no-clear` (она не сохранится в schedule).
- Агент в Docker-runtime (`runtime: docker`) — там и так batch-режим, чистка no-op, но шаблон recall всё равно полезен (агент стартует с нуля каждый раз).
