## Описание

Класс `AgScheduler` уже есть (`scheduler/scheduler.py`, AGOS-0017), но в runtime не поднимается — `main.py.bootstrap()` его не создаёт и не вызывает `.start()`. То есть cron-записи в таблице `schedule` сейчас не исполняются, даже если их добавить руками. Эта таска закрывает дыру: шедулер поднимается в `bootstrap`, корректно шатдаунится при остановке бота, и появляется Telegram-интерфейс для управления задачами без SQL.

После этой фичи пользователь сможет сказать боту `/schedule_add 0 * * * * @master проверь новую почту` и получить живую cron-задачу.

## Acceptance Criteria

- [ ] `AgScheduler` создаётся в `bootstrap()` и стартует через `await scheduler.start()` до запуска бота
- [ ] Grace-shutdown: при остановке `run_bot` / `run_all` вызывается `scheduler.stop()` (finally-блок)
- [ ] `create_bot` принимает опциональный `scheduler: AgScheduler | None`; `main.py` его пробрасывает
- [ ] Telegram-команда `/schedule_list` — вывод всех задач из БД (id, cron, agent, prompt, last_run, last_result)
- [ ] Telegram-команда `/schedule_add <min> <hour> <day> <month> <dow> @agent <prompt>` — добавляет задачу через `scheduler.add_task(...)`, новая задача сразу регистрируется в живом APScheduler (метод это уже умеет)
- [ ] Telegram-команда `/schedule_rm <id>` — удаляет задачу
- [ ] Telegram-команда `/schedule_run <id>` — немедленное исполнение задачи (ручной триггер)
- [ ] Все команды — только для `allowed_users` из `TelegramConfig`
- [ ] Валидация cron-выражения: при ошибке — понятное сообщение, не падение
- [ ] Валидация агента: если `@agent` не существует — отказ, не создавать «битую» задачу
- [ ] Секция в `docs/quick-start.md` с юзкейсом «почта раз в час» и ссылкой из оглавления
- [ ] Все тесты проходят

## Затрагиваемые модули

- `main.py` — bootstrap поднимает scheduler, run_bot/run_all стартуют и останавливают его
- `telegram/bot.py` — `create_bot(scheduler=...)`, регистрация новых CommandHandler-ов
- `telegram/handlers.py` — четыре новых `handle_schedule_*` хендлера
- `docs/quick-start.md` — новая секция «Планировщик: cron-задачи для агентов»
- `tests/test_handlers.py` — тесты команд с mock-scheduler
- `tests/test_scheduler.py` — тест live-регистрации через `add_task` при запущенном scheduler (если ещё не покрыт)

## Ключевые интерфейсы

```python
# main.py
async def bootstrap(config_path) -> tuple[AgentManager, Database, AppConfig, AgScheduler]:
    ...
    scheduler = AgScheduler(db=db, agent_manager=manager)
    await scheduler.start()
    return manager, db, config, scheduler

async def run_bot(config_path):
    manager, db, config, scheduler = await bootstrap(config_path)
    guard = _build_guard(config.guard, manager, db)
    app = create_bot(config.telegram, manager, guard=guard, scheduler=scheduler)
    try:
        await app.run_polling()
    finally:
        scheduler.stop()
```

```python
# telegram/handlers.py
async def handle_schedule_add(update, context, manager, scheduler, allowed): ...
async def handle_schedule_list(update, context, scheduler, allowed): ...
async def handle_schedule_rm(update, context, scheduler, allowed): ...
async def handle_schedule_run(update, context, scheduler, allowed): ...
```

## Edge Cases

- Пользователь вне whitelist → молча игнорируем (как остальные хендлеры)
- Невалидный cron → reply с описанием ошибки APScheduler
- `@agent` не существует в БД → reply «агент не найден», запись не создаётся
- Prompt пустой → reply «промт не может быть пустым»
- `scheduler.stop()` на уже остановленном → оборачиваем в try/except
- Задача падает с исключением в `_execute_task` → уже обрабатывается в AgScheduler, но проверить что logging видим в консоли бота

## План реализации

### Step 1: bootstrap возвращает scheduler

Изменить сигнатуру `bootstrap()` → возвращать четвёртым элементом `AgScheduler`. Обновить `run_bot`, `run_tui`, `run_all` — передавать/игнорировать.

### Step 2: create_bot принимает scheduler

Добавить параметр в `create_bot(scheduler: AgScheduler | None = None)`. Регистрируем новые CommandHandler только если передан.

### Step 3: Новые handlers

Четыре функции в `telegram/handlers.py`. Использовать существующий паттерн `allowed_users` check. Парсинг аргументов через `context.args`.

### Step 4: Тесты

- `test_handlers.py`: mock `AgScheduler`, проверка что `/schedule_add` с валидным cron зовёт `scheduler.add_task(...)` с правильными аргументами; с невалидным cron — отвечает пользователю без падения.
- `test_scheduler.py`: проверка что `add_task` при `scheduler.running == True` вызывает `_register_job` (возможно уже есть, дополнить).
