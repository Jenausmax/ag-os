> **Статус: идея, не в плане реализации.**
> Эта фича зафиксирована для будущего рассмотрения. Не начинать работу без явного переназначения статуса в `pending` и повторного брейншторма скоупа — к моменту, когда до неё дойдут руки, контекст может существенно измениться.

## Мотивация

После AGOS-0028 мастер управляет AG-OS через bash-вызовы `python main.py agent create ...` и парсинг stdout/JSON. Это работает, но имеет ограничения:

- Bash-tool у Claude Code — универсальный инструмент, а не типизированный API. Ошибки в аргументах ловятся только после запуска команды.
- Парсинг JSON-вывода — лишний этап для модели, съедает токены.
- Нет встроенной схемы tool-вызовов, которые модель могла бы планировать на уровне tool-use, а не на уровне «соберу bash-строку».
- Нет разграничения прав между командами — мастер через bash может сделать что угодно, не только команды AG-OS.

MCP-сервер `ag-os-control` даёт Claude Code типизированный набор тулов, подключаемый через `~/.claude/settings.json`. Мастер получает примерно то же, что и через CLI, но на уровне протокола tool-use: `create_agent(name, runtime, provider)` → структурированный ответ → дальнейшее планирование.

## Примерный скоуп (если дойдёт до реализации)

- Поднять MCP-сервер на Python (`mcp` SDK или `fastmcp`)
- Выставить тулы: `create_agent`, `destroy_agent`, `list_agents`, `add_schedule`, `list_schedules`, `remove_schedule`, `remember`, `recall`, `reload_schedule`
- Под капотом — те же методы `AgentManager` / `AgScheduler` / `MemorySystem`, что использует CLI из AGOS-0028
- Whitelist тулов в конфиге AG-OS: какие команды разрешены мастеру, какие — permanent-агентам, какие — dynamic
- Audit-log всех вызовов в отдельную таблицу `mcp_audit`
- Подключение к мастеру через `claude mcp add` в инсталляционном скрипте

## Почему отложено

- **CLI control-plane (AGOS-0028) покрывает 90% реальных кейсов** с заметно меньшей сложностью. До замера, где именно CLI упирается в потолок, вкладываться в MCP преждевременно.
- **MCP SDK активно развивается.** Делать сейчас — рискуешь переписать через 3-6 месяцев под новые API. Подождать, пока интерфейс устаканится.
- **Whitelist прав — отдельный большой дизайн.** Без продуманной модели прав MCP превращается в тот же bash с лишним слоем.
- **Нет boundary-условий.** Пока не ясно, какие тулы реально нужны мастеру (это станет видно по логам AGOS-0028 — что именно он зовёт из CLI и как часто). Без этих данных скоуп MCP — угадайка.

## Что сделать, прежде чем возвращаться к этой идее

1. Дождаться 1-2 месяцев продуктивной эксплуатации AGOS-0028 в проде.
2. Собрать статистику: какие CLI-команды мастер зовёт чаще всего, какие ошибки парсинга случаются, где он «тупит».
3. По этим данным — решить, даст ли MCP реальное улучшение UX/токенов/ошибок, или это just-for-fun.
4. Если даст — открыть новый брейншторм на актуальной версии MCP SDK и пересобрать скоуп.
