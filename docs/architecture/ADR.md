## ADR-001: Python как основной язык

**Статус:** Accepted

**Контекст:** Нужен язык для оркестратора, который работает с tmux, Docker, Telegram API, TUI и LLM API. Основные кандидаты: Python, TypeScript, Go.

**Решение:** Python 3.11+

**Альтернативы:**
- **TypeScript/Node.js** — хорошая экосистема для ботов, но слабая поддержка TUI и tmux
- **Go** — высокая производительность, но бедная экосистема для LLM SDK и TUI

**Последствия:**
- (+) Богатая экосистема: python-telegram-bot, Textual, libtmux, docker-py, anthropic SDK
- (+) Async/await из коробки (asyncio)
- (+) Быстрая итерация и прототипирование
- (-) Производительность ниже Go/Rust, но для оркестратора это некритично
- (-) GIL — но async I/O решает проблему для нашего use case

## ADR-002: tmux как менеджер процессов для хостовых агентов

**Статус:** Accepted

**Контекст:** Нужен способ запускать несколько CLI-агентов (Claude Code) параллельно, отправлять им команды и читать вывод. Агенты — интерактивные CLI-процессы.

**Решение:** tmux — одна сессия `ag-os`, каждый агент в отдельном окне.

**Альтернативы:**
- **subprocess (stdin/stdout)** — сложно управлять интерактивными процессами, парсинг ненадёжен
- **screen** — менее функционален, устаревший API
- **Собственный PTY-менеджер** — сложность разработки без явных преимуществ

**Последствия:**
- (+) Готовый менеджер процессов, не нужен IPC
- (+) `send_keys` / `capture_pane` — простой интерфейс для общения с агентами
- (+) Можно подключиться к агенту напрямую через `tmux attach` для отладки
- (+) libtmux — Python-обёртка с хорошим API
- (-) Зависимость от tmux на хосте
- (-) `capture_pane` имеет ограничения по буферу (решается параметром `-S`)

## ADR-003: SQLite как хранилище данных

**Статус:** Accepted

**Контекст:** Нужно хранить реестр агентов, память, расписание, логи guard. Данные локальные, одна инстанция приложения.

**Решение:** SQLite через aiosqlite (асинхронный доступ).

**Альтернативы:**
- **PostgreSQL** — избыточен для одного инстанса, требует отдельного сервера
- **Redis** — хорош для кэша, но не для структурированных данных и долговременного хранения
- **JSON-файлы** — нет транзакций, сложно с конкурентным доступом

**Последствия:**
- (+) Zero-config, файловая БД
- (+) ACID-транзакции
- (+) Достаточная производительность для нашей нагрузки
- (+) aiosqlite для async-совместимости
- (-) Одновременный доступ на запись — один writer (для нас ОК, single-instance)
- (-) Нет репликации (для нас некритично)

## ADR-004: Textual как TUI-фреймворк

**Статус:** Accepted

**Контекст:** Нужен терминальный дашборд с таблицами, горячими клавишами, live-обновлением. Должен быть async-совместимым.

**Решение:** Textual (Python)

**Альтернативы:**
- **Rich + ручной рендеринг** — низкоуровневый, много boilerplate
- **curses/urwid** — устаревшие, сложный API
- **blessed/npyscreen** — менее активная разработка

**Последствия:**
- (+) Современный, красивый, CSS-подобная стилизация
- (+) Async из коробки
- (+) DataTable, экраны, биндинги — всё встроено
- (+) Активная разработка (Textualize)
- (-) Относительно новый, API может меняться
- (-) Тяжелее минималистичных решений

## ADR-005: Иерархическая модель памяти

**Статус:** Accepted

**Контекст:** Агентам нужна память для сохранения контекста между сессиями. При этом мастер должен видеть всё, а агенты — только релевантное.

**Решение:** Три уровня видимости: private, shared, global. Мастер видит все записи.

**Альтернативы:**
- **Общая память (flat)** — все видят всё. Проще, но утечка контекста между агентами
- **Полностью изолированная** — каждый агент видит только своё. Нет возможности делиться знаниями
- **Pub/sub модель** — агенты подписываются на топики. Сложнее реализовать

**Последствия:**
- (+) Мастер как центр знаний — видит контекст всех агентов
- (+) Гибкое расшаривание через `share()`
- (+) TTL предотвращает накопление мусора
- (-) Дополнительная логика проверки доступа при каждом recall
- (-) Мастер может стать bottleneck если агенты часто обращаются к памяти

## ADR-006: Двухуровневая проверка промтов (regex + LLM)

**Статус:** Accepted

**Контекст:** Промты от пользователя (и потенциально от других агентов) должны проверяться на injection, опасные команды, утечку секретов.

**Решение:** Двухуровневый pipeline: быстрый regex (~1ms) → LLM Haiku (~200ms).

**Альтернативы:**
- **Только regex** — быстро, но легко обходится перефразированием
- **Только LLM** — ловит всё, но медленно и дорого для каждого промта
- **Embedding-based classifier** — средняя скорость, нужны обучающие данные

**Последствия:**
- (+) Regex отсекает очевидное мгновенно (нулевая задержка)
- (+) LLM ловит сложные/замаскированные атаки
- (+) При ошибке LLM API — промт помечается как SUSPICIOUS (fail-safe)
- (-) Расходы на API Haiku (~$0.001 за проверку)
- (-) Latency 200ms на каждый промт при включённом LLM-фильтре
- (-) Regex-правила нужно поддерживать вручную

## ADR-007: Docker для изоляции агентов

**Статус:** Accepted

**Контекст:** Некоторые агенты могут выполнять недоверенный код или работать с непроверенными LLM. Нужна изоляция.

**Решение:** Опциональный Docker runtime. При создании агента выбирается runtime: host (tmux) или docker.

**Альтернативы:**
- **Все агенты в Docker** — максимальная изоляция, но overhead для мастера и доверенных агентов
- **Все на хосте** — просто, но нет изоляции
- **nsjail/firejail** — легковеснее Docker, но менее стандартизированы

**Последствия:**
- (+) Гибкость: доверенные агенты на хосте (быстрее), недоверенные в Docker
- (+) Resource limits из коробки (CPU, RAM, сеть)
- (+) Изолированные volumes — агент видит только свой workspace
- (+) Docker-образы можно версионировать и воспроизводить
- (-) Overhead Docker на каждый контейнер (~50-100MB RAM)
- (-) Сложнее отладка Docker-агентов
- (-) docker-py добавляет зависимость

## ADR-008: Claude Code CLI как основной агент (подписка)

**Статус:** Accepted

**Контекст:** Нужен основной AI-агент. Варианты: Claude API (прямые вызовы), Claude Code CLI (через подписку), другие LLM.

**Решение:** Claude Code CLI как мастер-агент. Работает через подписку, не API — экономия.

**Альтернативы:**
- **Claude API напрямую** — проще интегрировать, но дорого ($15/M input, $75/M output для Opus)
- **Claude Agent SDK** — привязан только к Claude, нет мульти-LLM
- **LangChain/CrewAI** — мульти-LLM, но тяжёлые фреймворки

**Последствия:**
- (+) $20/месяц подписка вместо сотен $ на API
- (+) Claude Code уже имеет tools, skills, MCP, CLAUDE.md
- (+) CLI запускается в tmux — простая интеграция
- (-) CLI не предназначен для headless-режима — парсинг вывода ненадёжен
- (-) Привязка к интерактивному процессу, не к API
- (-) Rate limits подписки могут ограничивать параллельную работу

## ADR-009: python-telegram-bot как библиотека для Telegram

**Статус:** Accepted

**Контекст:** Нужна библиотека для Telegram Bot API с поддержкой inline-кнопок, polling/webhook, async.

**Решение:** python-telegram-bot v21+

**Альтернативы:**
- **aiogram** — тоже async, чуть быстрее, но менее документирована
- **telethon** — для user-аккаунтов, избыточен для бота
- **pyrogram** — MTProto, избыточен

**Последствия:**
- (+) Самая популярная Python-библиотека для Telegram-ботов
- (+) Async из коробки (v21+)
- (+) Встроенная поддержка InlineKeyboard для подтверждений
- (+) Хорошая документация и сообщество
- (-) Большая библиотека, много абстракций

## ADR-010: APScheduler как планировщик

**Статус:** Accepted

**Контекст:** Нужен планировщик для cron-задач. Должен работать в том же процессе (in-process), поддерживать cron-выражения и персистентность.

**Решение:** APScheduler 3.x с AsyncIOScheduler

**Альтернативы:**
- **Celery + Redis/RabbitMQ** — распределённый, избыточен для single-instance
- **crontab системы** — нет интеграции с Python-процессом, сложно управлять
- **schedule (pip)** — простой, но нет cron-синтаксиса и персистентности

**Последствия:**
- (+) In-process, не нужны внешние брокеры
- (+) CronTrigger с полным cron-синтаксисом
- (+) AsyncIOScheduler для async-совместимости
- (+) Можно сохранять job store в SQLite
- (-) APScheduler 4.x в разработке, API может измениться
- (-) Нет распределённого выполнения (для нас неактуально)
